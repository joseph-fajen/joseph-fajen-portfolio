## AI Documentation Principles I've Developed

<AccordionGroup>
  <Accordion title="Document limitations explicitly">
    AI systems fail in ways that are hard to predict. Good AI docs include a dedicated
    limitations section — what the system won't do, where it's unreliable, and what
    to do when it's wrong.
  </Accordion>
  <Accordion title="Separate the model from the product">
    Users don't care which LLM is underneath. Docs should describe behavior, not architecture —
    unless the architecture directly affects how users should work with the system.
  </Accordion>
  <Accordion title="Prompt engineering deserves documentation">
    For AI assistants, the prompts are part of the product. I document them — including
    the reasoning behind design choices — so they can be maintained and improved.
  </Accordion>
  <Accordion title="Test against adversarial inputs">
    Documentation accuracy for AI systems requires adversarial testing, not just
    happy-path validation. I design test queries to find the edges.
  </Accordion>
</AccordionGroup>
