## Who I write for

Developers don't read docs the same way end users do. They scan, they copy-paste, they trust accurate code samples more than prose. I write developer documentation with that in mind — structured for fast comprehension, validated against real implementations.

## Approach to developer docs

<AccordionGroup>
  <Accordion title="Code samples that actually run">
    I test code samples against real environments. If a code block is in a guide I wrote, it worked
    when I wrote it.
  </Accordion>
  <Accordion title="Audience-aware structure">
    Different developers need different entry points. I use explicit persona-based IA so
    a senior engineer and a first-timer can both find what they need without friction.
  </Accordion>
  <Accordion title="Collaborative by default">
    I work in the engineers' tools (Git, PRs, code review) and stay close to the implementation.
    Documentation written in isolation is documentation that goes stale.
  </Accordion>
</AccordionGroup>
